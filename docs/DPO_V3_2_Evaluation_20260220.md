# DPO 現状分析と次期計画 (V3.2) の妥当性評価
**作成日**: 2026-02-20

## 1. 現在のミッションと制約
*   **目標**: モデルに対し「挨拶・解説・Markdown (` ```json ` 等)」を一切出力させず、純粋なデータ（JSON等）のみをいきなり出力させること。
*   **制約条件**: コンペのレギュレーション上、推論時の「プロンプト変更（システムプロンプト等での口止め）」は不可。モデルの「重み」の調整のみで沈黙を実現しなければならない。

## 2. これまでの検証結果まとめ
1.  **SFT v8.5 (親モデル)**: 知能（正解率）は高いが、常にお喋り（Markdown + 挨拶）をする。
2.  **DPO V3 / V3.1**: 「短期集中・高学習率・お喋り拒絶データ」で学習させたが、標準プロンプトでの推論時にはお喋りが治らなかった。
3.  **【重要】プロンプト制御の効果**: SFT v8.5 や未学習の Qwen ベースモデルに対し、「絶対に喋るな」とプロンプトで指示すると、**完璧に黙って純粋な JSON を出力する** ことが判明した。モデル自体には「黙る（フォーマットに従う）能力」は備わっている。
4.  **【重要】LoRAスケーリングの逆効果**: DPO V3.1 のアダプタ適用強度を上げたところ、お喋りが減るどころか**「激化（解説が数段落に及ぶ）」**した。

## 3. なぜ DPO は効かなかったのか？（原因分析）
1.  **SFTデータとの強烈な矛盾**:
    *   直近の SFT 第8世代（v8.5）の学習データには、Markdown (` ```json ` 等) を含む回答が多数含まれていた（推測）。
    *   SFTで「Markdown が正解」と深く教え込んだ直後に、DPOで「Markdown はダメ」と教えているため、モデル内で重みが矛盾し、混乱を招いている。
    *   LoRA強度を上げて「お喋りが激化」したのも、この矛盾に耐えきれずモデルが「解説」で補おうとした結果（ハルシネーションの一種）と考えられる。
2.  **ベースモデルの「礼儀正しさ」の壁**:
    *   Qwen-Instruct は、標準的な指示（`Please convert...`）に対して丁寧に返答するように強固に事前学習されている。数百件の DPO では、その根深い「思考ルート」を上書きできなかった。

## 4. V3.2 計画 (Pure JSON DPO) の妥当性検討
**V3.2 計画案**: XMLやYAMLを排除し「純粋な JSON (Markdownなし)」のみを Chosen にした 200件の厳選データで DPO をやり直す。

### 【評価】
*   **メリット**: 学習データのノイズ（JSON以外の形式）が消えるため、モデルが「どの形式で出すべきか」で混乱しなくなる。
*   **懸念点・リスク**: 
    *   SFT段階での「Markdown＝正解」という刷り込みを少量のDPOで覆せるかは不透明。
    *   JSONだけに特化しすぎると、XML や YAML を要求された問題でエラーを起こす「過学習」のリスクが高まる。

## 5. 結論と代替案
V3.2 (Pure JSON DPO) は試す価値がありますが、**「SFT の段階ですでに Markdown 等を正解として覚えさせてしまった」**という根本原因を放置したままでは、再び対症療法に終わり、失敗する確率が高いと評価します。

### 【極めて確実な代替案：Silence SFT (SFT V9)】
DPO という「後からの矯正（劇薬）」に頼るのではなく、**最初から「お喋りなし・Markdownなし」の正解データのみを用いて SFT をやり直す** のが、最も素直で確実な解決策です。
1.  既存の強力な SFT データ（1万件等）の `output` 側から、プログラムで一括して Markdown (` ``` `) や挨拶文を全削除し、「純粋なデータだけ」にクレンジングする。
2.  その「無口なデータセット」を使って、SFT（V9）を実行する。
3.  モデルは「最初から黙ってデータだけを出すのがルールだ」と素直に学習するため、DPO のような副作用（混乱によるお喋り激化等）が起こらない。

## 6. ネクストアクション（提案）
DPO によるパッチ当て（V3.2等）を一旦保留し、**「SFT 学習データから Markdown/挨拶を消去するスクリプトの作成」** に着手することを強く推奨します。急がば回れですが、これがコンペ制覇（重みのみでの完全制御）の最適解だと考えます。
